# Stage 2: Cheap-to-Expensive Cascade

Stage 2 implements a two-stage physical operator for SUQL-style `answer(review, question)` filters:

1. A cheap model scores candidate rows as binary `1`/`0`.
2. A small expensive-model calibration sample learns a confidence cutoff.
3. Confident positive cheap scores are accepted early.
4. Confident negative cheap scores are rejected early.
5. Unsure scores are routed to the expensive full `answer()` model.

This is a local analogue of the Stretto idea of exposing a ladder of physical semantic-operator implementations. Here the ladder is:

```text
cheap score -> learn confidence cutoff -> early decision when confident -> expensive model when unsure
```

## Files

- `profiler.py`: legacy labelled-data threshold fitter.
- `cascade_filter.py`: runtime `answer()` implementation used by `src_baseline_stage2`.
- `calibrate.py`: legacy CLI for producing `thresholds.json`.
- `thresholds.json`: legacy fixed-threshold file retained for older experiments.
- `benchmark_stage2.py`: baseline vs Stage 2 benchmark driver.

## Configuration

Environment variables:

```bash
export SUQL_API_BASE="http://127.0.0.1:11434"
export SUQL_CHEAP_MODEL="ollama/gemma2:2b"
export SUQL_EXPENSIVE_MODEL="ollama/phi4-mini"
export SUQL_MODEL="$SUQL_EXPENSIVE_MODEL"
```

`SUQL_MODEL` is still used by the baseline and parser. Stage 2 uses `SUQL_CHEAP_MODEL` for the scoring step and `SUQL_EXPENSIVE_MODEL` for unsure fallback answers.
Model names can be written either as Ollama names (`gemma2:2b`) or LiteLLM Ollama provider names (`ollama/gemma2:2b`); the runtime normalizes them at each API boundary.

Make sure both models are installed locally, for example:

```bash
ollama pull gemma2:2b
ollama pull phi4-mini
```

`gemma2:2b` is the default cheap model because the saved Stage 2 experiments showed it is a better Ollama scorer for this workload than `llama3.2:1b`. The system can still fall back to `phi4-mini` for uncertain rows, which matters when a query may require sequential calls over hundreds of candidate reviews.

Runtime routing now follows the heterogen/BARGAIN-style cascade: the cheap score sign is the proxy label, `abs(score)` is confidence, and each `answer()` predicate learns the routing cutoff from an expensive-model calibration sample.

Optional routing controls:

```bash
export SUQL_CASCADE_TARGET="0.9"
export SUQL_CALIBRATION_BUDGET="20"
# Optional: bypass calibration and route with a fixed confidence cutoff.
export SUQL_MANUAL_CONFIDENCE_THRESHOLD="2.0"
```

## Try Cheap Models

Before running the full benchmark, smoke-test alternative cheap models against
the same binary scorer used by Stage 2:

```bash
python Stage_2/sweep_cheap_models.py \
  --api-base "$SUQL_API_BASE" \
  --installed-only \
  --models gemma2:2b llama3.2:1b smollm2:360m smollm2:1.7b tinyllama:1.1b
```

To run Stage 2 benchmarks for only the models whose scorer works:

```bash
python Stage_2/sweep_cheap_models.py \
  --api-base "$SUQL_API_BASE" \
  --installed-only \
  --run-benchmarks \
  --expensive-model "$SUQL_EXPENSIVE_MODEL" \
  --models gemma2:2b llama3.2:1b smollm2:360m smollm2:1.7b tinyllama:1.1b
```

The sweep writes logs and CSV results under:

```text
Stage_2/model_sweeps/
Stage_2/benchmarks/<sweep-run-name>_<model>/
```

## Legacy Labelled Calibration

Use a labelled CSV with columns:

```text
review,question,label
```

where `label` is `1/0` or `Yes/No`.

```bash
python Stage_2/calibrate.py labelled_examples.csv \
  --output Stage_2/thresholds.json \
  --cheap-model "$SUQL_CHEAP_MODEL" \
  --api-base "$SUQL_API_BASE"
```

The profiler fits one accept threshold and one reject threshold per question using Bayesian lower bounds on decision precision. This file-based mode is retained for older experiments; the current Stage 2 runtime learns a BARGAIN-style confidence cutoff online from the current candidate set.

## Benchmark

```bash
python Stage_2/benchmark_stage2.py \
  --sample-size 100 \
  --seed 11 \
  --api-base "$SUQL_API_BASE" \
  --model "$SUQL_EXPENSIVE_MODEL" \
  --cheap-model "$SUQL_CHEAP_MODEL" \
  --cascade-target 0.9 \
  --calibration-budget 20
```

Benchmark plots are generated by the shared plotter and can be regenerated with
custom implementations or metrics:

```bash
python scripts/plot_benchmarks.py per-question \
  --metrics-csv Stage_2/benchmarks/<run-name>/metrics.csv \
  --impls "baseline stage2" \
  --metrics "wall_seconds engine_seconds cheap_score_calls expensive_full_calls result_rows" \
  --output Stage_2/benchmarks/<run-name>/comparison_plot.png
```

For sample-size plots over existing `experiment_#1_*` runs:

```bash
python scripts/plot_benchmarks.py scaling \
  --benchmark-dir Stage_2/benchmarks \
  --pattern "experiment_#1_*/metrics.csv" \
  --sample-regex "experiment_#1_([0-9]+)" \
  --impls "baseline stage2" \
  --metrics "wall_seconds engine_seconds llm_prompts_issued expensive_full_calls cheap_score_calls result_rows" \
  --output-dir Stage_2/benchmarks/experiment_#1_analysis_custom
```

Outputs are written under:

```text
Stage_2/benchmarks/<run-name>/
```
